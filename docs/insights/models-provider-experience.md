# Models / Providers 体验收敛

对应交接文档：[handover/models-provider-experience.md](../handover/models-provider-experience.md)

## 背景

Models 页现在变复杂，不是因为用户真的需要这么多能力，而是因为每轮修复都在页面上继续加按钮、badge、tooltip 和兼容路径。结果是：

- 用户不知道“添加服务商”和“添加模型”分别该去哪。
- 用户不知道“刷新模型”到底是在服务商页做，还是模型页做。
- 页面上出现大量技术状态：catalog、upstream、enable source、Runtime、兼容性、旧 SKU、不可用、刷新失败。
- 很多状态不是用户能行动的事实，只是我们内部迁移和调试留下来的痕迹。

这已经偏离原始目标：**让用户连接服务商、选择模型，然后开始聊天。**

## 用户视角的目标

用户打开 Settings 时只需要理解三件事：

1. 我连接了哪些服务商。
2. 我现在能用哪些模型。
3. 我缺模型时怎么添加。

Models 页也只应该回答三个问题：

1. 默认会用哪个模型。
2. 当前有哪些可用模型。
3. 如何添加一个新模型。

其它信息默认都不应该占主路径。

## 页面职责

### Providers 页

Providers 页负责“连接渠道”，不是管理模型。

保留：

- 添加 / 编辑 / 删除服务商。
- API Key / OAuth / Base URL / 套餐入口说明。
- 连接状态和必要错误提示。

降级或移除：

- 每张服务商卡片上的“管理模型”入口。
- 每张服务商卡片上的常驻“刷新模型”按钮。
- 与模型默认值、模型启用隐藏、Runtime 兼容性相关的设置。

用户连接服务商后，如果该服务商有确定模型来源，就由系统给出默认模型；如果需要补模型，引导到 Models 页的“添加模型”。

### Models 页

Models 页负责“选择和补充模型”，不是展示所有同步细节。

默认可见三段：

1. **默认模型**：Auto / Pinned，以及当前会解析到的 Provider · Model。
2. **可用模型列表**：只展示当前对用户有意义的模型。
3. **添加模型**：一个统一入口，根据 provider 类型切换不同添加方式。

高级操作默认收起：

- 刷新 / 校验上游。
- 整理早期导入目录。
- 同步日志。
- 旧 SKU / 不在推荐目录说明。

## 模型拉取规则

### 原则

能不拉就不拉。模型同步不是用户的日常动作。

只有两种情况需要主动尝试拉取：

1. 用户刚新增 / 修改 API Key。
2. 用户刚新增 / 修改 API 请求地址。

如果服务商本身不支持可靠拉取模型，就不要显示“刷新模型”按钮。

### 分类

| 类型 | 示例 | 默认行为 | 用户添加 |
|------|------|----------|----------|
| 套餐型 / CodePlan | 火山、百炼、GLM、MiniMax、Kimi、小米 MiMo | 使用我们维护的套餐白名单 | “添加模型”补 SKU |
| 大目录聚合 | OpenRouter | 不全量落库，只保留必要默认别名 | 搜索并添加 |
| 普通可发现服务商 | 部分 OpenAI-compatible / 本地服务 | 新增或配置变更后尝试拉取 | 搜索 / 手动添加 |
| 不可发现服务商 | 固定目录、无模型列表 API、文档不稳定 | 不显示刷新模型 | 使用内置模型或手动添加 |

## 添加模型规则

“添加模型”应该是一条统一路径，而不是多个分散按钮。

用户点击添加模型后：

- OpenRouter：搜索远程目录，选择后添加。
- 套餐型服务商：默认展示我们维护的模型；如果用户要补控制台里可见的 SKU，也走搜索/输入添加。
- 普通服务商：先尝试拉取可搜索列表；拉不到时提供手动输入 model id。

如果用户不添加，就继续使用我们维护的默认模型列表。

## 该砍掉或降级的东西

- 常驻“刷新全部”按钮。
- Provider 卡片上的“管理模型”按钮。
- Provider 卡片上的常驻“刷新模型”按钮。
- 无真实后端验证的“可用”标签。
- 用户无法行动的 `enable_source` / `catalog` / `upstream` 细节。
- 默认展开的不可用模型。
- 为迁移期遗留数据单独铺开的主路径 UI。
- 行级堆叠标签，例如“API 同步 / 手动启用 / 默认 / 已不再推荐”同时出现在一个模型后面。
- “按推荐整理”这类迁移期清理功能作为主路径入口。

这些内容不是不能存在，而是只能进“高级 / 详情 / 更多”。

## 进一步收敛判定（2026-05-06）

本轮发现：仅把“刷新全部 / 按推荐整理”折叠进高级区，实际简化非常有限。真正让页面难懂的是模型行仍在展示过多内部来源标签，且很多高级操作本身就是迁移期产物。

### 模型行只保留用户能行动的信息

模型行默认只显示：

1. 模型显示名。
2. 模型 ID 或短 ID。
3. 默认标记（仅当前默认模型显示）。
4. 不可用提示（仅当当前执行引擎确实不可用，且用户需要处理时显示）。

默认隐藏：

- API 同步。
- 手动启用。
- enable_source。
- source。
- catalog / upstream。
- 已不再推荐。

这些信息如果仍有调试价值，只能放在“更多 / 详情 / 高级信息”里，不能占据每一行模型的主视觉。

### “自动”模式必须透明

Auto 不能只显示“自动”。用户需要知道自动当前会解析到哪一个模型。

默认区域至少显示：

- 模式：自动。
- 当前会用：服务商 · 模型。
- 如当前解析结果不可用，显示原因和修复入口。

否则用户仍然会被迫切回手动固定模型。

### “按推荐整理”默认不再作为功能目标

“按推荐整理”本质是迁移期清理，不是普通用户日常需求。更好的产品逻辑是：

- 我们维护好推荐列表和 CodePlan 官方表。
- 用户自己添加 / 映射的模型不自动整理。
- 旧数据清理可以保留为隐藏的维护工具，但不进入主路径，也不作为 Phase 1 的用户价值目标。

### “刷新全部”默认不再保留

如果单个 provider 不再常驻“刷新模型”，全局“刷新全部”也不应该常驻。

未来只有两种刷新入口：

1. 用户更新 API Key / Base URL 后，系统自动尝试一次。
2. 高级维护区里的“重新检测模型来源”，且只对真正支持模型列表 API 的 provider 生效。

### OpenRouter / Opencode 特殊说明

OpenRouter 这类渠道不应该让用户看到“大目录整理”作为日常动作。

- “整理早期导入的目录”只用于旧版本导入过 300+ 模型的迁移场景，默认不展示。
- “当前执行引擎不可用”必须解释清楚：是当前执行引擎不支持这个 provider，还是角色映射只对 Claude Code 引擎生效。
- 角色映射入口必须明确写明：这是 Claude Code 引擎使用的别名映射，不等于所有执行引擎都会使用。

## 术语统一

中文界面不要直接展示 `Runtime`。

建议：

- Runtime → 执行引擎。
- Claude Code Runtime → Claude Code 引擎。
- AI SDK / CodePlan Runtime → CodePilot 引擎。
- CodePlan → 套餐模型 / CodePlan 套餐，按场景选择。
- Provider → 服务商。
- Model → 模型。

同一个概念不要在不同页面叫不同名字。

## 验收问题

后续每次改 Models / Providers 页，都先问：

1. 用户能不能在 5 秒内知道该去哪添加服务商？
2. 用户能不能在 5 秒内知道该去哪添加模型？
3. 页面上是否只有 2-3 个主动作？
4. 这个 badge 是否对应一个真实、可解释、用户能处理的状态？
5. 这个“刷新”是否真的会改变用户需要的结果？
6. 这个术语是不是用户能理解，而不是工程内部词？

如果答案是否定的，先砍或降级，不要继续加解释。
