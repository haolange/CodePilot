## 引言
本文件系统性阐述本仓库中的流式响应处理能力，重点围绕服务器推送事件（SSE）的实现原理、连接管理、数据流解析与缓冲、渲染与回调分发、错误重连与超时恢复策略，并对比传统请求-响应模式的优势。文档同时提供可操作的调试与性能监控建议，帮助开发者在前端与后端协同场景中稳定地消费流式数据。

**更新** 本次更新重点关注中断和停止功能的重大改进，包括会话锁定结算系统和聊天API路由中的令牌会计处理，以及强制中止策略的实现。

## 项目结构
与流式响应直接相关的核心模块分布如下：
- 钩子层：负责消费浏览器/运行时的 ReadableStream 并按行解析 SSE，分发到上层回调
- 解析器层：兼容旧版 Claude 的 SSE 行格式解析
- 会话管理：对长连接进行生命周期管理、超时与重试策略，包含新的会话锁定结算系统
- 安全与代理：对流进行安全转换与代理适配
- 中断与停止：实现强制中止和会话锁定管理
- 令牌会计：在聊天API路由中处理令牌使用统计
- 测试用例：覆盖 SSE 帧格式、事件类型与文件变更事件的契约

## 核心组件
- 流式消费与解析
  - 使用浏览器/运行时提供的 ReadableStreamDefaultReader 迭代读取二进制块，按字节解码为字符串，按行切分，识别以"data: "开头的数据行，解析为 JSON 事件对象，分发到上层回调。
  - 通过闭包与 ref 包装回调，避免闭包捕获陈旧状态，确保 onText/onToolUse 等回调始终绑定最新逻辑。
- SSE 行格式兼容
  - 兼容旧版 Claude 的 SSE 行格式解析，保证事件类型与数据结构的一致性。
- 会话生命周期与超时
  - 对长连接设置总时长与空闲时长双重超时，超时后优雅结束；支持在会话管理器层面进行重连与退避策略。
  - **新增** 会话锁定结算系统，防止死锁情况下的资源泄漏。
- 中断与停止功能
  - 实现强制中止策略，即使中断请求失败也不会影响流的正常终止。
  - 支持优雅中断和强制中止两种模式。
- 令牌会计处理
  - 在聊天API路由中实现精确的令牌使用统计和会计处理。
  - 支持错误路径的令牌会计，确保不会遗漏任何使用情况。
- 安全与代理
  - 提供安全流封装与 Codex 代理流转换，确保跨域或中间层代理场景下的稳定性与安全性。
- 测试保障
  - 单元测试覆盖 SSE 行级消费、SDK 固件帧格式以及特定事件（如文件变更）的契约，确保行为稳定。

## 详细组件分析

### 组件一：SSE 流消费与回调分发（useSSEStream）
- 职责
  - 消费 ReadableStream 的二进制块，解码为文本，按行切分，过滤非 data 行，解析 JSON 事件并分发到回调。
  - 通过 ref 包装回调，避免闭包捕获陈旧状态，确保 onText/onToolUse 等回调始终绑定最新逻辑。
  - 维护累积文本与令牌用量等上下文，最终返回处理结果。
- 关键流程
  - 读取 -> 解码 -> 行切分 -> 过滤 data 行 -> JSON 解析 -> 分发回调 -> 更新上下文 -> 循环
- 错误处理
  - 对于无法解析的行，跳过并继续处理后续行，保证整体流不中断。
- 性能要点
  - 使用流式解码减少内存占用；按行处理避免一次性解析大块数据。
  - 通过 ref 包装回调，避免频繁重建函数导致的额外开销。

```mermaid
flowchart TD
Start(["开始"]) --> Read["读取下一个块"]
Read --> Done{"是否 done?"}
Done --> |是| Return["返回累积结果"]
Done --> |否| Decode["按流式解码为字符串"]
Decode --> Split["按行切分，保留最后一行作为缓冲"]
Split --> Loop["遍历每一行"]
Loop --> Check{"是否以 data: 开头?"}
Check --> |否| Next["跳过该行"] --> Loop
Check --> |是| Parse["JSON 解析事件"]
Parse --> Dispatch["分发到回调 onText/onToolUse/onResult 等"]
Dispatch --> Accumulate["累积文本/更新用量"]
Accumulate --> Loop
Loop --> |循环结束| Read
```

### 组件四：会话锁定结算系统（session-lock-settle）
- 职责
  - 作为共享的结算器，确保会话锁定在正常完成和强制中止两种路径下都能正确释放。
  - 实现幂等性保证，防止重复释放和状态翻转。
- 核心机制
  - **幂等性**：只执行第一次调用的副作用，后续调用成为无操作
  - **强制清理**：总是停止续期间隔，确保不会出现永久续期的情况
  - **所有权检查**：只有在仍然拥有锁的情况下才写入运行时状态
- 安全保证
  - 锁释放是按锁ID作用域的（删除匹配该锁ID的行），防止释放过期锁
  - 运行时状态写入是按会话作用域的，防止覆盖新请求的状态

```mermaid
flowchart TD

Check --> |否| Mark["标记为已结算"]

Clear --> Release["释放会话锁"]
Release --> Owned{"是否仍拥有锁?"}
Owned --> |是| Status["写入运行时状态"]
Owned --> |否| Skip["跳过状态写入"]

NoOp --> End
```

## 依赖关系分析

- 外部依赖
